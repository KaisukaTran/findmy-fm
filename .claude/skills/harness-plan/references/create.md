# create subcommand — plan-creation flow

Interview for ideas and requirements, then generate an executable Plans.md.

## Step 0: Check the conversation context

When requirements can be extracted from the prior conversation, confirm them:

> Choose how to build the plan:
> 1. From the prior conversation — build the plan based on the brainstorm
> 2. From scratch — start from an interview

For "From the prior conversation": extract the requirements, ideas, and decisions and confirm with the user.
After confirming, skip to Step 3 (technical research).

## Step 1: Ask what to build

If there is no user input, ask:

> What do you want to build?
>
> e.g. reservation management system / blog site / task management app / API server
>
> A rough idea is fine!

## Step 2: Increase resolution (max 3 questions)

> Tell me a bit more:
>
> 1. Who will use it? (just you? a team? public?)
> 2. Is there a service you want to reference?
> 3. How far do you want to build? (MVP? full features?)

## Step 3: Planning-quality check

Details: `references/planning-quality.md`

Do not drop the information the user gave you straight into Plans.md.
When it includes an external product, competitor, spec proposal, improvement idea, or comparison material, vet it with the latest information, existing specs, memory, and multi-perspective TeamAgent / subagent review, and turn only the elements worth adopting into the task contract.
Any planning that is not a one-off, trivial task is treated as requiring TeamAgent or subagents.

Minimum checks:

- latest information: prefer WebSearch / official docs / primary sources, and confirm key points across multiple sources
- existing specs: check Plans.md, README, docs, CLAUDE.md, related skills, and tests
- memory: when harness-mem / harness-recall / `.claude/agent-memory/` / `.claude/state/` are available, check them project-scoped to avoid reinventing the wheel
- discussion: separate adoption value and risk from the Product / Architecture / Security / QA / Skeptic perspectives
- quality foundation: for a plan with source code changes, check the `formatter_baseline` and, if lint / formatter is unset, run a setup task first
- implementation-plan validation: check product fit, security fit, and works-in-practice, and land test / smoke / CI / review / release gates into the DoD
- scoring: rate Product Fit, Evidence Strength, User Value, Implementation Feasibility, Regression Safety, Strategic Leverage, Security Safety, and Works In Practice out of 5

Do not read the `harness-mem` DB directly. When search or a documented memory surface is unavailable, state "memory not checked" explicitly.
When the Task tool is unavailable, state `subagents not used` explicitly and evaluate the same perspectives separately on your own.
Output `team_validation_mode` as one of `not_required_lightweight` / `native` / `subagent` / `manual-pass` / `unavailable`.
For non-trivial planning, use one of `native` / `subagent` / `manual-pass`, and do not mark something Required while it is still `unavailable`.
Product / Architecture / Security / QA / Skeptic are perspective names, not agent_type names.
The Security gate does not require actually reading a `.env` or a secret.

For small typos, formatting, README/CHANGELOG, or marker-only updates, this step may be done lightly.

## Step 4: Technical research (WebSearch)

Do not ask the user; Claude Code researches and proposes.

```
WebSearch:
- "{{project type}} tech stack 2025"
- "{{similar service}} architecture"
```

## Step 4.4: spec.md / Plans.md two-source-of-truth check

Plans.md is the task contract that fixes "what to do".
The root `spec.md` is the product contract that fixes "what is correct".
Do not mix the two. `/harness-plan create` is not a Plans.md-generation command; treat it as a surface that returns the co-required planning output of the spec.md product contract and the Plans.md task contract.
Keep the precedence `spec.md > sub-spec > Plans.md`.

The output of `/harness-plan create` must always be this 2-item set:

1. a `Spec delta` or a `Spec skip reason`
2. generated `Plans.md` tasks

Read the root `spec.md` every time. When an implementation decision might drift, update the root `spec.md` before creating Plans.md.
Do not make the user write the spec from scratch. The `Spec delta` / `Spec skip reason` is generated by Harness; the consumer only approves or edits it.
The agent drafts the minimal delta from the existing spec, repo evidence, memory, tests, and the input requirements, and only surfaces options when the decision is split.

### Conditions to create/update the spec source of truth

- user-visible behavior is added or changed
- it decides API, data model, permissions, billing, external integrations, or tenant boundary
- there are multiple implementation proposals where the choice changes product behavior
- a past or current conversation shows implementation drift due to spec ambiguity
- Plans.md has tasks but the project-level correctness conditions are not documented

### Conditions you may skip

- typos / formatting / lint only
- dependency bumps only
- README / CHANGELOG only
- docs-only / mechanical task
- a narrow refactor with no behavior change
- the correct answer is clear from existing specs and tests

Even when skipping, do not omit the `Spec skip reason`.
Even for a docs-only / mechanical task, leave the skip reason in the task context / sprint contract.

### Destination

The top priority is the root `spec.md`.
Only when the consumer repo has no root `spec.md`, update an existing project-level spec as a fallback.
If there is neither a root `spec.md` nor an existing project spec, create:

```text
docs/spec/00-project-spec.md
```

The first spec may be short. At minimum, include Purpose, Users And Workflows, Core Rules, Data And Contracts, Non-Goals, Open Decisions, and Links.

Details: `docs/plans/spec-ssot.md`

## Step 4.6: lint / formatter baseline check

For a plan with source code changes, check the lint / formatter baseline before creating implementation tasks.
This is not a "cleanup" task; it is a gate to set up, in advance, a foundation for verifying quality as Yes/No after implementation.

What to check:

- JavaScript / TypeScript: the `lint` / `format` scripts in `package.json`, and ESLint / Prettier / Biome / Oxlint / dprint config or dependency
- Python: Ruff / Black / isort / mypy config in `pyproject.toml`
- Go: `gofmt` / `go test` / `go vet` / lint-equivalent CI command
- Rust: `cargo fmt` / `cargo clippy` / `cargo test`
- existing CI: quality commands in `.github/workflows`, `scripts/ci/*`, `Makefile`, etc.

Leave a `formatter_baseline` in the output:

```text
formatter_baseline: configured | missing | not_applicable | unknown
formatter_baseline_evidence: [files / commands seen]
formatter_baseline_action: none | add_setup_task | skip_with_reason | spike
```

If it is unset and there are source code changes, add a setup task before the implementation tasks in Plans.md.
The setup task's DoD is "config / script / validation command are in place, and a broad bulk reformat is explicitly out of scope".
Do not install packages during planning. The introduction work is executed by harness-work as a setup task.

Conditions you may skip:

- docs-only / markdown-only / changelog-only
- there are existing lint / formatter / CI commands that sufficiently cover the languages touched by this change
- introduction is impossible due to consumer-repo constraints. Even so, leave a `formatter_baseline_action: spike` or a skip reason

## Step 5: Extract the feature list

Extract a concrete feature list from the requirements.

Example: a reservation management system
- user registration / login
- reservation calendar view
- create / edit / cancel a reservation
- admin dashboard
- email notifications
- payment

## Step 5.5: Generate an optional brief

Attach a brief only when needed. A brief does not replace Plans.md; it is a short supporting document that fixes the implementation premises.

- for a task that includes UI, a `design brief`
- for a task that includes API, a `contract brief`
- when UI and API are mixed, split the briefs

### design brief

A brief for a UI task should include at minimum:

- what you want to achieve
- who uses it
- important screen states
- look-and-feel constraints
- completion conditions

### contract brief

A brief for an API task should include at minimum:

- what it receives / returns
- input-validation conditions
- failure-mode behavior
- external dependencies
- completion conditions

## Step 6: Build the priority matrix (2-axis evaluation)

Evaluate each feature on the two axes **Impact × Risk (risk/uncertainty)**:

- **Impact**: user value × number of target users (high/low)
- **Risk**: technical unknown × external dependency (high/low)

| Impact ＼ Risk | low risk | high risk |
|-------------|---------|---------|
| **high Impact** | ★ **Required** — top priority (value is certain) | ▲ **Required + [needs-spike]** — needs early validation |
| **low Impact** | ○ **Recommended** — do it with spare capacity | ✕ **Optional** — defer or reduce scope |

### `[needs-spike]` marker

Automatically attach the `[needs-spike]` marker to high-Impact × high-Risk tasks.
For tasks tagged `[needs-spike]`, automatically generate a **spike (technical validation) task** and place it first:

```markdown
| N.X-spike | [spike] technical validation of {{task name}} | validation report produced | - | cc:TODO |
| N.X       | {{task name}} [needs-spike] | {{DoD}} | N.X-spike | cc:TODO |
```

The spike task's completion condition is "leave a validation report (feasible / infeasible / needs design change)".

## Step 6.5: TDD skip decision (default enabled)

TDD is enabled by default. Attach a `[skip:tdd]` marker and skip only for tasks matching one of these:

| Skip condition | Reason |
|-------------|------|
| documentation/comments only | does not affect executable code |
| config files only (JSON, YAML, .env) | no logic under test |
| a one-line-or-less simple fix (typo) | test cost exceeds the benefit |
| style/format change only | does not affect behavior |
| dependency update only | no implementation-logic change |
| README/CHANGELOG update | documentation only |
| refactoring (no behavior change) | already covered by existing tests |

Tasks not matching the above get TDD applied automatically (test-first recommended).

## Step 6.7: Plans.md v3 format spec

Plans.md v3 includes the following format extensions:

### Purpose line in the Phase header (optional)

Each Phase header can carry a one-line Purpose. Omit it when there is no input:

```markdown
### Phase N.X: [phase name] [Px]

Purpose: [the problem this phase solves, in one line]
```

- **default**: do not ask for input (omit when blank)
- **effect when present**: shown in the breezing Phase 0 scope check
- **generation rule**: auto-write only when the user explicitly states the phase purpose

### Artifact notation (Status column)

On task completion, attach the commit hash to Status:

```markdown
| Task | Content | DoD | Depends | Status |
|------|------|-----|---------|--------|
| 1.1  | ... | ... | - | cc:done [a1b2c3d] |
| 1.2  | ... | ... | 1.1 | cc:TODO |
```

- **format**: `cc:done [7-char hash]`
- **timing**: auto-attached in `harness-work` Solo Step 7
- **backward compatible**: `cc:done` without a hash is still valid

### Affected-file list

Files related to the v3 format:

| File | Effect |
|---------|------|
| `skills/harness-plan/references/create.md` | add the Purpose line to the Step 6 template |
| `skills/harness-plan/references/sync.md` | recognize the `cc:done [hash]` form in diff detection |
| `skills/harness-work/SKILL.md` | attach the hash in Solo Step 7, re-ticket on failure |
| `skills/harness-sync/SKILL.md` | save a snapshot with --snapshot |
| `skills/breezing/SKILL.md` | show progress in the Progress Feed |

## Step 7: Generate Plans.md

First output a `Spec delta` or `Spec skip reason`, then auto-generate the quality markers + DoD + Depends and produce Plans.md.

### Spec result output

The `Spec delta` / `Spec skip reason` is generated by Harness; the consumer only approves or edits it.

```markdown
Spec delta:
- path: spec.md
- change: [the product rule to add/change]
- why: [why it is needed as a premise of this task contract]

Plans.md:
| Task | Content | DoD | Depends | Status |
|------|------|-----|---------|--------|
```

```markdown
Spec skip reason:
- path checked: spec.md
- reason: [docs-only / mechanical task / correct answer already fixed by existing specs and tests]
- preserve in: task context or sprint contract

Plans.md:
| Task | Content | DoD | Depends | Status |
|------|------|-----|---------|--------|
```

### Quality-marker attachment logic
```
analyze the task content
    ↓
├── "auth" "login" "API" → [feature:security]
├── "component" "UI" "screen" → [feature:a11y]
├── "fix" "bug" → [bugfix:reproduce-first]
├── "docs" "comment" "README" "CHANGELOG" → [skip:tdd]
├── "config" "json" "yaml" "env" → [skip:tdd]
├── "style" "format" "lint" → [skip:tdd]
├── "refactor" (no behavior change) → [skip:tdd]
├── "payment" "billing" → [feature:security]
└── otherwise → no marker (TDD enabled by default)
```

### DoD auto-inference logic

Infer the DoD from the task "content" by keyword and auto-fill it:

| Keyword in task content | DoD inference |
|---------------------|---------|
| "create" "new" "add" | the file exists and has the expected structure |
| "test" | tests pass (`npm test` / `pytest` etc.) |
| "fix" "bug" | the problem no longer reproduces |
| "UI" "screen" "component" | visual confirmation (screenshot or browser) |
| "API" "endpoint" | response confirmed with curl/httpie |
| "config" | the config value takes effect |
| "docs" "documentation" | the file exists with no broken links |
| "migration" "DB" | migration is runnable |
| "refactoring" | all existing tests pass + 0 lint errors |

The inferred result is only a default. When the user specifies concrete acceptance conditions, prefer those.

### Depends auto-inference logic

Infer inter-task dependencies within a phase by these rules:

1. **DB/schema tasks** → depended on by other implementation tasks (predecessor)
2. **UI tasks** → depend on API/logic tasks (successor)
3. **test/validation tasks** → depend on implementation tasks (last)
4. **config/environment tasks** → depended on by other tasks (predecessor)
5. **tasks with no clear dependency** → `-` (parallelizable)

When unsure of the inference, set `-` and ask the user to confirm.

**Generation template**:

```markdown
# [Project name] Plans.md

Created: YYYY-MM-DD

---

## Phase 1: [phase name]

Purpose: [phase purpose (optional)]

| Task | Content | DoD | Depends | Status |
|------|------|-----|---------|--------|
| 1.1  | [task description] [feature:security] | [verifiable completion condition] | - | cc:TODO |
| 1.2  | [task description] | [verifiable completion condition] | 1.1 | cc:TODO |
```

**Purpose line**:
- auto-write only when the user states the phase purpose
- if there is no input, omit the whole Purpose line (do not leave a blank line)
- keep it to one line (no multiple lines)

**DoD (Definition of Done) notation**:
- write a verifiable one-liner (e.g. "tests pass", "migration runnable", "0 lint errors")
- "feels good" / "works fine" are forbidden. Make it judgeable as Yes/No

**Depends notation**:
- no dependency: `-`
- single dependency: a task number (e.g. `1.1`)
- multiple dependencies: comma-separated (e.g. `1.1, 1.2`)
- phase dependency: a phase number (e.g. `Phase 1`)

### Team mode output

Only when the user explicitly states team mode, also guide the issue-bridge dry-run separately from Plans.md.

- only one tracking issue
- list a sub-issue payload per task
- keep Plans.md as the source of truth
- guide the `scripts/plans-issue-bridge.sh --team-mode` dry-run in a form usable as-is

## Step 8: Always guide the session launch command and the first input

Right after producing Plans.md, so the user does not hesitate on the next step,
provide as a set **the launch command for a new session** and
**the first input to enter right after launch**.

### Rules for presenting

1. write at least one set of a concrete launch command + first input
2. if possible, narrow to "the top pick + one alternative"
3. add a one-line reason for why that combination, not just the command
4. for a long-running task, guide `bash scripts/claude-longrun.sh` first

### Recommended mapping

| Situation | Launch command | First input |
|------|--------------|------------|
| start from the first task | `claude` | a single task run like `/harness-work 1.1` |
| advance multiple tasks together | `claude` | `/breezing all` |
| advance everything serially | `claude` | `/harness-work all` |
| long-running / re-entry assumed | `bash scripts/claude-longrun.sh` | `/harness-loop all` |

### Output example

```text
Next step:
- Launch command for the new session: claude
- First input after launch: /breezing all
- Best suited for: this Plans.md advances multiple tasks together, so team execution is most natural
```

```text
Next step:
- Launch command for the new session: bash scripts/claude-longrun.sh
- First input after launch: /harness-loop all
- Best suited for: a long-running task likely to involve waits over 5 minutes or a resume
```

## Step 9: Next-action guidance

> Plans.md complete!
>
> Next steps:
> - start implementing with `harness-work`
> - or say "start from Phase 1"
> - add a feature with `harness-plan add [feature name]`
> - defer a feature with `harness-plan update [task] blocked`

## CI mode (--ci)

No interview. Use the existing Plans.md as-is and only break it down into tasks.

1. Load Plans.md
2. List the cc:TODO tasks in priority order
3. Mark parallelizable tasks with `[P]`
4. Propose the next task to run
