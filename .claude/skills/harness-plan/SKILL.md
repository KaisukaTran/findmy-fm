# Harness Plan

Harness unified planning skill. It merges three former skills:

- `planning` (plan-with-agent) — idea → Plans.md
- `plans-management` — task state management and marker updates
- `sync-status` — sync check between Plans.md and the implementation

## Quick Reference

| User input | Subcommand | Action |
|------------|------------|------|
| "create a plan" / `/harness-plan create` | `create` | Spec delta / skip reason → generate Plans.md tasks |
| "add a task" / `/harness-plan add` | `add` | Add a new task to Plans.md |
| "mark it done" / `/harness-plan update` | `update` | Change the task marker to `cc:done` |
| "where are we?" / `/harness-plan sync` | `sync` | Reconcile and sync the implementation with Plans.md |
| `/harness-sync` | `sync` | Progress check (equivalent to the standalone sync surface) |
| `/harness-plan create` | `create` | Create a plan as the two-source-of-truth spec.md / Plans.md |
| `/harness-plan list` | `list` | List the named Plans in `plans/manifest.json` |
| `/harness-plan switch <name>` | `switch` | Save the active plan to `.claude/state/active-plan.json` |

## Literal companion commands (CC 2.1.108+)

- `/recap`: when returning after a while, re-take a summary before entering `sync`
- `/undo`: alias for `/rewind`. Use directly when you want to immediately revert the last plan update

## Subcommand details

### Standard planning-quality contract

See [references/planning-quality.md](${CLAUDE_SKILL_DIR}/references/planning-quality.md)

`harness-plan` is the planning surface that produces the co-required planning output of the spec.md product contract and the Plans.md task contract.
Keep the precedence `spec.md > sub-spec > Plans.md`.
Plans.md is the task ledger and the root `spec.md` is the product contract; do not break that hierarchy.
Do not drop the information you were given straight into Plans.md.
When creating a plan or adding a large task, check the latest information, existing specs, memory, and multi-perspective discussion via TeamAgent / subagents, and convert only the elements worth adopting into the task contract.
`/harness-plan create` returns a `Spec delta` (or `Spec skip reason`) together with generated `Plans.md` tasks.
The output must always include a `Spec delta` or a `Spec skip reason`.
The `Spec delta` / `Spec skip reason` is generated by Harness; the consumer only approves or edits it.

**Non-trivial planning gate**:

Any planning that is not a one-off, trivial task is treated as requiring TeamAgent or subagents.
Here non-trivial means a request that affects multiple tasks / multiple files / multiple sessions / product behavior / API / data model / permissions / billing / external integrations / distribution surface / security.
When the Task tool is available, run the independent perspectives Product / Architecture / Security / QA / Skeptic.
When it is not available, state `subagents not used` explicitly and evaluate the same perspectives separately on your own.

Non-trivial planning output must always include the following validation:

- `team_validation_mode`: `not_required_lightweight` / `native` / `subagent` / `manual-pass` / `unavailable`
- consistency across `spec.md` / sub-spec / `Plans.md`
- a wheel-reinvention check via harness-mem / harness-recall / repo memory
- whether it stays aligned with the product purpose
- whether there are issues with security, permissions, secrets, or supply chain
- whether a lint / formatter baseline exists. For a plan with source code changes, if it is unset, place a setup task before the implementation tasks
- whether it is a workable plan — i.e. whether test / smoke / CI / review / release gates land in the task DoD

Lightweight tasks may use `team_validation_mode: not_required_lightweight`.
Non-trivial planning must use one of `native` / `subagent` / `manual-pass`.
Do not mark something Required while it is still `unavailable`.
Product / Architecture / Security / QA / Skeptic are validation perspectives, not agent_type names.
Hand them as perspectives to the available TeamAgent / Task subagents; do not require spawning arbitrary agents.
The Security gate does not require actually reading secrets.
If reading a `.env` or a secret becomes necessary, stop as a Risk Gate and confirm via permitted existing guards / evidence.

**When to apply**:

- creating a new plan with `create`
- adding a task with `add` that affects product behavior / API / permissions / billing / external integrations / distribution surface
- the user handed over an external product, competitor, spec proposal, improvement idea, or comparison material
- there is a risk of conflict with existing specs or past decisions

**When you may treat it lightly**:

- a marker-only `update`
- a status-only `sync`
- typos, formatting, README/CHANGELOG only
- a narrow change whose correct answer is already fixed by existing specs and tests

**Quality flow**:
1. Decompose the input information and make explicit what is being evaluated, the scoring axes, and the uncertain facts
2. Fetch the latest information. For external facts, prefer WebSearch / official docs / primary sources, and cross-check key points across multiple sources
3. Check existing specs, the root `spec.md`, Plans.md, README, docs, CLAUDE.md, and related skills
4. Check available memory surfaces project-scoped: harness-mem / harness-recall / `.claude/agent-memory/` / `.claude/state/`
5. For non-trivial planning, use TeamAgent / Task subagents and review independently from different perspectives such as Product / Architecture / Security / QA / Skeptic
6. For a plan with source code changes, check the lint / formatter baseline and, if unset, run a setup task first
7. Produce a neutral scoring review and classify into Required / Recommended / Optional / Reject
8. In the `$easy` format, report the proposal, the reasoning, and what will happen
9. Land only the adopted proposals into the root `spec.md` / Plans.md / test tasks

### create — create a plan

See [references/create.md](${CLAUDE_SKILL_DIR}/references/create.md)

Interview for ideas and requirements, then generate an executable Plans.md.

**Flow**:
1. Check the conversation context (extract from the prior discussion, or start a fresh interview)
2. Ask what to build (max 3 questions)
3. **Planning-quality check** (latest information, existing specs, memory, multi-perspective TeamAgent / subagent review, scoring)
4. Technical research (WebSearch)
5. Extract the feature list
6. **spec.md / Plans.md two-source-of-truth check** (Spec delta or Spec skip reason + Plans.md tasks)
7. Priority matrix (Required / Recommended / Optional / Reject)
8. TDD adoption decision (test design)
9. Generate Plans.md (with `cc:TODO` markers)
10. Next-action guidance

### spec.md / Plans.md two-source-of-truth check (default)

Treat Plans.md as the task contract for "what to do" and the root `spec.md` as the product contract for "what is correct".
Co-required planning output means both outputs are mandatory; the precedence stays `spec.md > sub-spec > Plans.md`.
When the implementation might drift, update the root `spec.md` before generating Plans.md.
`create` and product-impacting `add` read the root `spec.md` every time.

Preferred destination:

1. root `spec.md`
2. only when the consumer repo has no root `spec.md`: an existing project spec / architecture / product compass
3. only when the consumer repo has no root `spec.md`: `docs/spec/00-project-spec.md`
4. in a repo with existing conventions, a spec path that follows those conventions

Conditions that require creating/updating:

- a task that decides user-visible behavior, API, data model, permissions, billing, or external integrations
- a task with multiple implementation approaches where the choice changes product behavior
- a task with signs from a past or current conversation that "the spec was ambiguous and the implementation drifted"
- a task with work in Plans.md but no stable document of the project-level correctness conditions

Conditions where it is unnecessary:

- typos, formatting, dependency bumps, README/CHANGELOG only
- a narrow refactor with no behavior change
- a fix whose correct answer is sufficiently fixed by existing specs and tests

Output contract:

- `Spec delta`: when updating the product contract, write the target spec path and the changes
- `Spec skip reason`: when not updating the product contract, write the reason
- the `Spec delta` / `Spec skip reason` is generated by Harness; the consumer only approves or edits it
- even for a docs-only / mechanical task, leave a `Spec skip reason` in the task context / sprint contract
- do not conclude that a missing search result, unavailable memory, or unread file is absent. `not_observed != absent`
- do not make the user write the spec from scratch. The agent drafts the minimal delta from the existing spec and the input, and only surfaces decision branches when ambiguous

Reference:

- `docs/plans/spec-ssot.md`

### Session-launch guidance on create completion (required)

When `create` finishes, do not stop at an explanation — provide, as a set, **the launch command for a new session** and **the first instruction prompt to enter right after launch**.

Priority order:

1. there is only one incomplete task, or it is natural to start only the first one
   - launch command: `claude`
   - first input: `/harness-work <task number>`
2. there are multiple loosely-dependent tasks and it is natural to advance them together
   - launch command: `claude`
   - first input: `/breezing all`
   - alternative: `/harness-work all`
3. long-running or re-entry is assumed
   - launch command: `ENABLE_PROMPT_CACHING_1H=1 claude`
   - first input: `/harness-loop all`
   - alternative: `/breezing all`

Include at least these 3 lines:

- `Launch command for the new session:`
- `First input after launch:`
- `Best suited for:`

Example:

```text
Launch command for the new session: claude
First input after launch: /breezing all
Best suited for: Phase 1 has multiple tasks, so advancing them together is more natural
```

When recommending a long-running mode, also note the Claude Code session launch command:

```text
Launch command for the new session: ENABLE_PROMPT_CACHING_1H=1 claude
First input after launch: /harness-loop all
Best suited for: long-running tasks with waits over 5 minutes or that span a resume
```

Notes:

- `scripts/claude-longrun.sh` is a development-helper script in this repository and is not distributed to the consumer environment after plugin install
- therefore, in consumer-facing guidance always prefer the single-line `ENABLE_PROMPT_CACHING_1H=1 claude`
- only when you want an equivalent wrapper during repository development, `bash scripts/claude-longrun.sh` may be used on a local checkout

**CI mode** (`--ci`):
No interview. Use the existing Plans.md as-is and only break it down into tasks.

### add — add a task

Add a new task to Plans.md.
For a product-impacting addition, follow the "spec.md / Plans.md two-source-of-truth check" above and also output a `Spec delta` or `Spec skip reason`.

```
/harness-plan add <task name>: <details> [--phase <phase number>]
```

The task is added with a `cc:TODO` marker.

### update — update a marker

Change a task's status marker.

```
/harness-plan update [task name|task number] [WIP|done|blocked]
```

Marker mapping:

| Command | Marker |
|---------|---------|
| `WIP` | `cc:WIP` |
| `done` | `cc:done` |
| `blocked` | `blocked` |
| `TODO` | `cc:TODO` |

### sync — sync progress

Reconcile the implementation status with Plans.md, detect diffs, and update.

See [references/sync.md](${CLAUDE_SKILL_DIR}/references/sync.md)

**Flow**:
1. Get the current Plans.md state
2. Detect the Plans.md format (v1: 3 columns / v2: 5 columns)
3. Get the implementation status from git status / git log
4. Check the agent trace (`.claude/state/agent-trace.jsonl`)
5. Detect diffs between Plans.md and the implementation
6. Propose automatic fixes for stale markers
7. Present the next action

**Retrospective** (default ON):
If there is at least one `cc:done` task, automatically run a retrospective.
Analyze estimation accuracy, blocker-cause patterns, and scope changes, and record the learnings.
Can be explicitly skipped with `sync --no-retro`.

### team mode / issue bridge

Keep Plans.md as the source of truth and use GitHub Issue integration only in opt-in team mode.

- in solo development, do not use the bridge
- team mode creates one tracking issue and generates a per-task sub-issue payload under it as a dry-run
- `scripts/plans-issue-bridge.sh` does not actually update GitHub; it always returns a dry-run payload
- this bridge makes no changes to Plans.md

Reference:

- `docs/plans/team-mode.md`

### named Plans

When using multiple Plans.md files, make `plans/manifest.json` the source of truth and select by name.

```bash
scripts/plan-registry.sh list
scripts/plan-registry.sh switch roadmap
scripts/plans-issue-bridge.sh --plan roadmap --format markdown
node scripts/generate-sprint-contract.js --plan roadmap 9.1.1
```

Operating rules:

- use only one named plan per run
- for long-running / CI / issue bridge, do not rely on the active pointer; pass `--plan <name>`
- manifest paths are relative to the project root only. Absolute paths, `..`, and symlinks outside the repo are rejected

## Plans.md format convention

### Format

```markdown
# [Project name] Plans.md

Created: YYYY-MM-DD

---

## Phase N: Phase name

| Task | Content | DoD | Depends | Status |
|------|------|-----|---------|--------|
| N.1  | description | tests pass | - | cc:TODO |
| N.2  | description | 0 lint errors | N.1 | cc:WIP |
| N.3  | description | migration runnable | N.1, N.2 | cc:done |
```

**DoD (Definition of Done)**: a one-line, verifiable completion condition. "feels good" / "works fine" are forbidden. Make it judgeable as Yes/No.

**Depends**: inter-task dependencies. `-` (no dependency), a task number (`N.1`), comma-separated (`N.1, N.2`), or a phase dependency (`Phase N`).

### TDD tags

A Plans.md task can carry a tag in its content or DoD that makes the TDD decision explicit.

| Tag | Meaning | `tdd_required` inference |
|------|------|--------------------|
| `[tdd:required]` | this task needs a failing test written first | `true` |
| `[tdd:skip:<reason>]` | this task skips TDD with a reason | `false`, `skip_tdd_reason=<reason>` |

`<reason>` must not be empty.
Examples: `[tdd:skip:docs-only]`, `[tdd:skip:no-test-framework-detected]`.

When there is no tag, infer `tdd_required` in this order:

1. Plans.md tag: `[tdd:required]` / `[tdd:skip:<reason>]`
2. files: if it includes source implementation such as `src/`, `app/`, `cmd/`, `lib/`, `pkg/`, `internal/`, `go/`, then required
3. TDD inference: for docs-only or no test framework, attach a skip reason and treat as not required

### optional briefs / manifest

`harness-plan create` attaches a brief only when needed.

- a project spec SSOT is the document that fixes the project-wide correctness conditions; create it only when needed
- for a task that includes UI, a `design brief`
- for a task that includes API, a `contract brief`
- a brief is a short supporting document that fixes "what to build"; it does not replace Plans.md or the spec SSOT
- the skill-frontmatter listing can be turned into machine-readable JSON with `scripts/generate-skill-manifest.sh`

Reference:

- `docs/plans/briefs-manifest.md`
- `docs/plans/spec-ssot.md`

### Marker list

| Marker | Meaning |
|---------|------|
| `pm:requested` | requested by the PM |
| `cc:TODO` | not started |
| `cc:WIP` | in progress |
| `cc:done` | worker work complete |
| `pm:reviewed` | PM review complete |
| `blocked` | blocked (always record the reason) |
